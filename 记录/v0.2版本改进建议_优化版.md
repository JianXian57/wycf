# 《我要吃饭》v0.2 版本改进建议（优化版）

## 一、总体说明

我已完成上一版本的手动验证。大部分功能运行正常，本轮主要需要修复以下问题：

1. 中文文本在页面跳转和保存后出现 URL 编码字符串；
2. “吃的什么”输入框初始渲染时文字和占位符被裁切；
3. 抽取结果区域的按钮语义和布局需要重新设计；
4. 最近抽取记录的状态标识和操作按钮需要优化；
5. 实际吃饭记录需要明确显示来源；
6. 记录页面需要同时提供“去选饭”和“去做饭”的快捷入口。

本轮修改必须保留现有本地数据。不得通过清空本地存储规避兼容问题。

---

## 二、修复中文 URL 编码显示问题

### 2.1 当前问题

从抽取结果点击“记入记录”进入吃饭记录编辑页后，“吃的什么”字段可能显示：

```text
%E9%A3%9F%E5%A0%82
```

而不是正常的中文：

```text
食堂
```

该异常文本有时可以正常保存，但保存后在“记录”页面仍然显示 URL 编码字符串。

### 2.2 修复要求

页面跳转、表单初始化、保存和列表展示的整个链路中，业务数据必须始终使用正常中文字符串。

优先采用以下方案：

- 页面跳转只传递抽取记录 ID，例如 `drawId`；
- 目标页面根据 `drawId` 从统一 store 中读取抽取结果；
- 不要把完整中文名称直接拼接到页面 URL；
- 不要把 `encodeURIComponent` 的结果保存到本地数据中。

如果现有页面结构仍必须通过 URL 参数传递中文：

- 发送端只编码一次；
- 接收端只解码一次；
- 解码后再写入页面 data；
- 保存前确保字段为正常中文；
- 避免重复编码和重复解码；
- 对 `decodeURIComponent` 使用异常保护。

### 2.3 已有异常数据修复

现有本地存储中可能已经存在 `%E9%A3...` 形式的数据，不能只修复新数据。

请增加安全的文本规范化或数据迁移逻辑：

- 检测字符串是否可能为合法的百分号编码 UTF-8 文本；
- 仅在可以安全解码时执行解码；
- 普通包含 `%` 的文字不得被误处理；
- 对选饭名称、菜谱名称、吃饭记录名称及相关快照进行必要检查；
- 成功迁移后保存正常中文；
- 提升 `schemaVersion`；
- 迁移失败时保留原始数据，不得清空用户数据。

## 三、修复“吃的什么”输入框初始显示不完整

### 3.1 当前问题

吃饭记录编辑页中的“吃的什么”输入框在页面刚打开时：

- 已填内容只显示上半部分；
- 未填写时的 placeholder 只显示下半部分；
- 点击输入框后才恢复正常显示。

### 3.2 修复要求

检查该控件的 WXML 和 WXSS，确保在页面初始渲染时：

- 中文内容完整显示；
- placeholder 完整显示；
- 文字垂直居中；
- 不依赖点击或聚焦后才修正布局；
- 不被父容器的 `overflow`、固定高度或错误的 `line-height` 裁切；
- 在微信开发者工具模拟器和真机中均正常。

如果该字段是单行内容，应优先使用原生 `input`，不要使用不必要的 `textarea`。

样式应明确设置并协调：

- `height`
- `line-height`
- `padding`
- `box-sizing`
- `font-size`
- 父容器高度
- 垂直对齐

不得通过给文字增加不稳定的顶部偏移来临时掩盖问题。

## 四、重新设计抽取结果区域的四个操作按钮

### 4.1 术语说明

为避免混淆，本轮统一使用以下术语：

- **抽取历史**：首页或抽取历史页中的“最近抽取记录”；
- **吃饭记录**：记录页面中用户实际吃了什么的记录。

“记下来”表示进入吃饭记录编辑页并在用户确认后创建吃饭记录。

### 4.2 四个按钮

在“本次选饭结果”和“本次做饭结果”区域中，设置以下四个按钮。

按钮顺序必须按两行两列排列，从左到右、从上到下依次为：

```text
第一行：
[ 好，就吃这个，记下来！ ] [ 不算，再抽一次！ ]

第二行：
[ 记一下，再抽个别的吧~ ] [ 算了，不抽了~ ]
```

小屏设备上必须保证：

- 两列等宽；
- 文案可以正常换行；
- 不溢出容器；
- 按钮之间有足够间距；
- 点击区域清晰；
- 不出现按钮互相挤压或文字贴边。

### 4.3 四个按钮的精确定义

#### A. 好，就吃这个，记下来！

- 保留当前抽取历史；
- 打开吃饭记录编辑页并自动预填当前结果；
- 用户确认保存后创建一条吃饭记录；
- 保存成功后不再自动抽取；
- 返回对应的选饭页或做饭页；
- 当前抽取历史显示为“已记录”。

如果用户在编辑页取消或直接返回：

- 不创建吃饭记录；
- 当前抽取历史仍然保留；
- 状态仍为“未记录”；
- 不自动继续抽取。

#### B. 不算，再抽一次！

- 不创建吃饭记录；
- 删除当前这一次抽取历史；
- 立即开始一次新的抽取。

实现时必须通过当前抽取记录的唯一 ID 精确删除，不能根据结果名称删除。

#### C. 记一下，再抽个别的吧~

- 保留当前抽取历史；
- 打开吃饭记录编辑页并自动预填当前结果；
- 用户确认保存后创建一条吃饭记录；
- 保存成功并返回后，自动开始一次新的抽取；
- 原抽取历史显示为“已记录”。

如果用户在编辑页取消或直接返回：

- 不创建吃饭记录；
- 当前抽取历史仍然保留；
- 状态仍为“未记录”；
- 不自动开始新的抽取。

必须通过明确的返回意图或页面状态实现“保存后继续抽取”，不能依赖不稳定的延时或重复触发 `onShow`。

#### D. 算了，不抽了~

- 不创建吃饭记录；
- 删除当前这一次抽取历史；
- 不再开始下一次抽取；
- 清除当前结果卡片或恢复为未抽取的初始状态；
- 页面保持可继续手动点击“开始抽取”的状态。

### 4.4 防重复要求

四个按钮都必须防止重复点击：

- 操作开始后立即禁用相关按钮；
- 不得重复创建吃饭记录；
- 不得重复删除同一条抽取历史；
- 不得同时启动多个抽取动画；
- 页面返回后正确恢复按钮状态。

## 五、增加“结果说明”弹窗

### 5.1 入口位置

在以下标题右侧增加一个紧凑的说明入口：

- “本次选饭结果”
- “本次做饭结果”

入口内容：

- 感叹号图标；
- 文字“结果说明”。

要求：

- 与标题在同一行；
- 位于标题右侧；
- 不使用过长的整行按钮；
- 点击区域足够；
- 图标和文字垂直居中。

### 5.2 弹窗内容

点击后弹出规则说明，推荐使用 `wx.showModal`，仅显示确认按钮。

```text
好，就吃这个，记下来！
创建吃饭记录，不再继续抽取。

不算，再抽一次！
不创建吃饭记录，删除本次抽取历史，并重新抽取。

记一下，再抽个别的吧~
创建吃饭记录，保存成功后继续抽取。

算了，不抽了~
不创建吃饭记录，删除本次抽取历史，也不再抽取。
```

弹窗文案必须明确区分：

- 是否创建吃饭记录；
- 是否保留抽取历史；
- 是否继续抽取。

抽取结果区域下方原有的长说明文字可以移除，避免页面拥挤。

## 七、优化最近抽取记录

### 7.1 状态文字

每条最近抽取记录右侧当前显示“选项”，语义不明确。

修改为：

- 尚未生成关联吃饭记录：`未记录`
- 已生成关联吃饭记录：`已记录`

状态应根据真实数据关联判断：

- 吃饭记录保存 `sourceDrawId`；
- 抽取记录状态通过是否存在对应 `sourceDrawId` 的吃饭记录计算；
- 如果关联的吃饭记录之后被删除，该抽取记录应重新显示为“未记录”。

手动创建、没有 `sourceDrawId` 的吃饭记录不影响任何抽取记录状态。

### 7.2 状态徽标样式

- 文字水平、垂直居中；
- 使用合适的左右内边距；
- 高度与字体匹配；
- 不强制为正圆形；
- 文案不换行；
- `已记录` 与 `未记录` 的视觉状态可区分，但整体保持简洁；
- 不使用会造成文字偏移的异常 `line-height` 或 transform。

### 7.3 操作按钮间距

每条最近抽取记录中的“记入记录”和“删除”按钮当前距离过近。

修改要求：

- 两个按钮之间留出明确间距；
- 点击区域不能重叠；
- 不与状态徽标挤在一起；
- 小屏设备上允许合理换行；
- 操作按钮与点击记录主体分离；
- 点击按钮时不得误触发菜谱详情跳转。

已记录的抽取历史：

- 将“记入记录”改为“已记入”并禁用，或隐藏该按钮；
- 不得重复生成关联吃饭记录。

## 八、完善实际吃饭记录的数据来源

每条吃饭记录增加并规范以下来源字段：

```js
{
  sourceType: "choice" | "recipe" | "manual",
  sourceDrawId: "draw_xxx" | null
}
```

含义：

- `choice`：由选饭结果创建；
- `recipe`：由做饭结果创建；
- `manual`：用户在记录页手动新增。

记录页面中的每条记录都应显示来源：

- `选饭`
- `做饭`
- `手动记录`

建议使用小型来源徽标，不要把来源混入“吃的什么”字段中。

已有数据兼容：

- 能通过 `sourceDrawId` 推断来源的，自动补全；
- 无法推断来源的旧记录默认标记为“手动记录”；
- 不得清空旧记录。

## 九、记录页面增加两个快捷入口

在记录页面“所有记录”区域右上角，当前只有“去选饭”。

修改为：

```text
去选饭    去做饭
```

行为：

- “去选饭”切换到底部“选饭”Tab；
- “去做饭”切换到底部“做饭”Tab；
- 使用适合 Tab 页面切换的微信小程序 API；
- 不要重复打开同一页面栈；
- 两个入口宽度由内容决定；
- 不使用整行长按钮；
- 在小屏设备上保持合理间距。

这里指“所有记录”区域标题右侧的快捷入口，不是在每一条记录中重复放置两个按钮。

本轮修改涉及：

- 中文异常数据修复；
- 吃饭记录来源字段；
- 抽取记录与吃饭记录关联；

必须：

- 保留现有 storage key；

- 增加幂等的数据迁移；
- 同一迁移重复运行不会破坏数据；
- 迁移前后数组顺序和时间戳保持不变；
- 不清空用户的选项、菜谱、抽取历史和吃饭记录；
- 迁移失败时记录错误并尽可能继续使用原数据；
- 不把编码异常字符串继续传播到新数据。
