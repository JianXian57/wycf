# 《我要吃饭》v0.3 版本改进建议（优化版）

## 一、总体说明

我已完成上一版本的手动验证。当前主要问题集中在：

1. 抽取结果在“记下来”后仍然停留，导致用户还能继续点击其他按钮，产生状态冲突；
2. “记一下”的产品语义与实际实现不一致；
3. 抽取结果区域的四个按钮尺寸和布局需要进一步优化；
4. 多处辅助按钮仍采用上下排列，不符合当前页面的信息密度；
5. “查看全部”入口仍然过大或位置不够紧凑；
6. 记录页中的快捷入口和操作按钮需要统一为横向布局。

本轮修改重点是明确四个结果按钮的最终状态、避免同一结果被重复处理，并统一页面中的横向按钮布局。

不得通过清空本地数据解决问题。

---

## 二、修复“记下来”后当前结果仍可继续操作的问题

### 2.1 当前问题

抽到结果后，点击：

```text
好，就吃这个，记下来！
```

吃饭记录可以正常创建，但“本次选饭结果”或“本次做饭结果”仍然保留在页面中。

此时用户还可以继续点击：

- “不算，再抽一次！”
- “算了，不抽了~”

如果继续点击，当前抽取历史可能被删除，但已经创建的吃饭记录仍然存在，导致抽取历史与实际吃饭记录之间出现不一致。

### 2.2 修复要求

“好，就吃这个，记下来！”成功保存吃饭记录后，该次结果必须进入终止状态。

具体要求：

- 保留该次抽取历史；
- 创建并保存对应的吃饭记录；
- 该次抽取历史状态显示为“已记录”；
- 返回选饭页或做饭页后，清除“本次结果”卡片；
- 四个结果操作按钮全部消失；
- 页面恢复为尚未抽取的初始状态；
- 不自动开始下一次抽取；
- 用户只能重新点击“开始抽取”产生新结果。

不得只把按钮设为禁用后继续保留旧结果，应当明确清空当前待处理结果。

### 2.3 防止二次操作

当前结果应当有明确的唯一 ID，例如：

```js
currentDrawId
```

当某个操作已经完成后：

- 立即锁定当前结果；
- 清空 `currentDrawId`；
- 后续点击事件即使被重复触发，也不得再次处理；
- 不得重复创建吃饭记录；
- 不得重复删除抽取历史；
- 不得继续操作已经完成的结果。

所有四个按钮都需要使用同一套防重复机制。

## 三、重新定义“暂记一下，再抽个别的吧~”

### 3.1 原有理解错误

上一版本中，“记一下，再抽个别的吧~”被实现为：

- 创建吃饭记录；
- 然后继续抽取。

这不符合我的实际意图。

### 3.2 正确含义

按钮文案修改为：

```text
暂记一下，再抽个别的吧~
```

它的含义是：

- 保留当前抽取历史；
- 不创建吃饭记录；
- 当前抽取历史保持“未记录”状态；
- 立即开始一次新的抽取；
- 新的抽取结果成为新的“本次结果”。

这里的“暂记一下”只表示：

> 将当前结果保留在最近抽取记录中，作为备选结果。

不表示用户已经决定吃它，也不表示要进入吃饭记录编辑页。

### 3.3 不需要页面跳转

点击“暂记一下，再抽个别的吧~”时：

- 不进入吃饭记录编辑页；
- 不创建 `mealRecord`；
- 不填写 `sourceDrawId`；
- 不修改当前抽取历史的“未记录”状态；
- 直接开始下一次抽取动画。

## 四、四个按钮的最终业务规则

本轮后，四个按钮的含义固定如下。

| 按钮 | 保留抽取历史 | 创建吃饭记录 | 继续抽取 | 操作后当前结果 |
|---|---:|---:|---:|---|
| 好，就吃这个，记下来！ | 是 | 是 | 否 | 清除 |
| 不算，再抽一次！ | 否 | 否 | 是 | 被新结果替换 |
| 暂记一下，再抽个别的吧~ | 是 | 否 | 是 | 被新结果替换 |
| 算了，不抽了~ | 否 | 否 | 否 | 清除 |

### 4.1 好，就吃这个，记下来！

- 保留当前抽取历史；
- 打开吃饭记录编辑页；
- 自动预填当前结果；
- 用户保存成功后创建吃饭记录；
- 抽取历史显示“已记录”；
- 返回后清除当前结果；
- 不继续抽取。

如果用户取消编辑或直接返回：

- 不创建吃饭记录；
- 当前抽取历史仍然保留；
- 状态仍为“未记录”；
- 当前结果仍然保留；
- 四个按钮继续可用。

### 4.2 不算，再抽一次！

- 删除当前抽取历史；
- 不创建吃饭记录；
- 立即开始新的抽取；
- 新结果替换当前结果。

### 4.3 暂记一下，再抽个别的吧~

- 保留当前抽取历史；
- 不创建吃饭记录；
- 不进入编辑页；
- 当前抽取历史仍显示“未记录”；
- 立即开始新的抽取；
- 新结果替换当前结果。

### 4.4 算了，不抽了~

- 删除当前抽取历史；
- 不创建吃饭记录；
- 不继续抽取；
- 清除当前结果；
- 页面恢复为初始状态。

## 五、同步修改“结果说明”

“结果说明”弹窗中的文案需要同步修改。

推荐文案：

```text
好，就吃这个，记下来！
创建吃饭记录，保留本次抽取历史，不再继续抽取。

不算，再抽一次！
删除本次抽取历史，不创建吃饭记录，并重新抽取。

暂记一下，再抽个别的吧~
保留本次抽取历史作为备选，不创建吃饭记录，并重新抽取。

算了，不抽了~
删除本次抽取历史，不创建吃饭记录，也不再抽取。
```

必须明确区分：

- 抽取历史是否保留；
- 吃饭记录是否创建；
- 是否继续抽取。

不得再将“暂记一下”解释为创建吃饭记录。

## 六、优化四按钮的尺寸和布局

### 6.1 当前问题

当前四个按钮虽然已经采用两行两列布局，但：

- 按钮整体偏大；
- 高度或内边距过多；
- 文案和按钮比例不协调；
- 页面视觉重心过重；
- 四个按钮之间的层级不够清晰。

### 6.2 布局要求

仍然保持两行两列：

```text
第一行：
[ 好，就吃这个，记下来！ ] [ 不算，再抽一次！ ]

第二行：
[ 暂记一下，再抽个别的吧~ ] [ 算了，不抽了~ ]
```

要求：

- 两列等宽；
- 行间距和列间距一致；
- 左右边距合理；
- 按钮整体比当前版本更紧凑；
- 文案允许两行显示；
- 按钮高度以完整容纳两行文字为准，不要额外留出过多空白；
- 字号可适当缩小，但仍需清晰可点击；
- 不允许文字溢出、被裁切或贴边；
- 常见小屏设备中保持两列，不改为单列；
- 选饭页和做饭页必须使用同一套样式。

### 6.3 视觉层级建议

可以通过轻微样式差异区分操作性质：

- “好，就吃这个，记下来！”作为主操作；
- “不算，再抽一次！”作为明显的重新抽取操作；
- “暂记一下，再抽个别的吧~”作为次要保留操作；
- “算了，不抽了~”作为弱化或中性操作。

保持整体温暖、简洁，不使用过强的警告色。

## 七、抽取区下方两个辅助按钮改为横向排列

### 7.1 涉及页面

选饭页：

- 去管理选项
- 查看抽取记录

做饭页：

- 去管理菜谱
- 查看抽取记录

### 7.2 修改要求

当前两个按钮为上下排列，修改为左右横向排列。

示意：

```text
[ 去管理选项 ]    [ 查看抽取记录 ]
```

或：

```text
[ 去管理菜谱 ]    [ 查看抽取记录 ]
```

要求：

- 两个按钮位于同一行；
- 左右分布；
- 宽度可以相等，或根据文案合理分配；
- 页面左右保留适当空白；
- 两个按钮之间有明确间距；
- 按钮尺寸比主抽取按钮更小；
- 不抢夺“开始抽取”按钮的视觉重点；
- 小屏设备上不溢出；
- 选饭页和做饭页样式一致。

建议使用一个统一的横向辅助操作容器，避免分别写大量局部样式。

## 八、优化“最近抽取记录”的“查看全部”入口

### 8.1 位置

“查看全部”必须位于“最近抽取记录”标题栏右上角，与标题处于同一行。

示意：

```text
最近抽取记录                         查看全部 >
```

### 8.2 样式

- 不使用占满整行的大按钮；
- 宽度由文字内容决定；
- 取消过大的固定宽度；
- 适当缩小内边距；
- 可以使用文字按钮或紧凑胶囊按钮；
- 视觉上弱于记录列表中的主要操作；
- 点击区域仍需足够；
- 标题和按钮垂直居中；
- 不因长标题或小屏设备发生重叠。

选饭页和做饭页保持一致。

## 十、每条吃饭记录的“编辑”和“删除”改为横向排列

### 10.1 当前问题

每条记录中的：

- 编辑

目前采用上下排列，占用过多高度。

### 10.2 修改要求

```text
[ 编辑 ]    [ 删除 ]
```

要求：

- 按钮尺寸紧凑；
- 保持足够点击区域；
- 两个按钮之间有明确间距；

- 点击操作按钮时不得误触记录卡片的其他事件。

建议所有记录卡片共用统一的操作栏样式。

## 十一、状态一致性要求

本轮需要重点处理页面状态一致性。

### 11.1 当前结果状态

应至少区分：

```js
idle
animating
pending
navigating
completed
```

含义：

- `idle`：没有当前结果；
- `animating`：抽取动画中；
- `pending`：当前结果等待用户处理；
- `navigating`：正在进入吃饭记录编辑页；
- `completed`：当前操作刚完成，正在清理状态。

不要求必须使用这些具体字符串，但必须建立等价的状态控制，避免多个操作同时发生。

### 11.2 页面返回处理

从吃饭记录编辑页返回时，需要明确判断：

- 是否保存成功；
- 是由哪个抽取结果进入；
- 是否需要清除当前结果；
- 是否只是用户取消返回。

不得仅依赖页面重新显示就默认认为已经保存。

推荐通过明确的成功标记、事件通道或共享状态传递结果。

## 十二、回归验证要求

### 12.1 “好，就吃这个，记下来！”

分别在选饭和做饭页面验证：

- 点击后进入编辑页；
- 保存后创建一条吃饭记录；
- 对应抽取历史显示“已记录”；

- 抽取历史不能被后续误删。

取消编辑时验证：

- 不创建吃饭记录；
- 当前结果仍保留；
- 抽取历史仍为“未记录”；
- 四按钮仍可使用。

### 12.2 “暂记一下，再抽个别的吧~”

分别验证：

- 不进入编辑页；
- 不创建吃饭记录；
- 当前抽取历史保留；
- 当前抽取历史显示“未记录”；
- 自动开始新的抽取；
- 新结果替换旧结果。

### 12.3 其他两个按钮

“不算，再抽一次！”：

- 删除当前抽取历史；
- 不创建吃饭记录；
- 自动重新抽取。

“算了，不抽了~”：

- 删除当前抽取历史；
- 不创建吃饭记录；

### 12.4 防重复

验证以下区域均为横向布局：

- 做饭页“去管理菜谱 / 查看抽取记录”；
- 记录页“去选饭 / 去做饭”；
- 每条记录“编辑 / 删除”。

同时验证：

- 四个结果按钮大小合理；
- 两行两列布局稳定；
- “查看全部”位于右上角且尺寸紧凑；
- 小屏设备无溢出；
- 按钮文字完整显示；
- 选饭页和做饭页风格一致。
